<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stripe webhook → payments table | Untrusted HTTP POST from Stripe; must verify HMAC before any DB write |
| Client → recordPaymentAction | Untrusted form data; Zod validates and server re-checks over-application |
| Square sourceId → Square API | sourceId is single-use browser token; server must not retry on Square errors (double-charge risk) |
| PWA → processSquarePaymentAction | PWA tech user passes sourceId + amount; server must validate against invoice balance |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Spoofing | /api/webhooks/stripe | mitigate | stripe.webhooks.constructEvent(rawBody, sig, STRIPE_WEBHOOK_SECRET) — reject 400 if verification fails; never call req.json() before verification |
| T-07-04 | Repudiation | payments.stripeEventId UNIQUE | mitigate | Duplicate stripe_event_id returns 200 idempotent no-op — events cannot be reprocessed (D-06) |
| T-07-05 | Tampering | recordPaymentAction over-application | mitigate | Server-side guard: SUM(allocations) <= amount + 0.001 before insert; client-side guard in Wave 3 is defense-in-depth only |
| T-07-06 | Information Disclosure | Stripe Payment Link tenant isolation | mitigate | Webhook handler verifies invoice.tenantId matches metadata.tenantId before writing payment — cross-tenant link sharing returns 400 |
| T-07-07 | Tampering | Square sourceId reuse | accept | Square API rejects reused sourceId; server should NOT retry on Square errors — let the error surface to the user |
| T-07-08 | Elevation of Privilege | paymentMethods admin-only writes | mitigate | createPaymentMethodAction and deletePaymentMethodAction (Wave 3) check orgRole === 'admin' via Clerk roles before withTenant write |
</threat_model>
