<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client form → createJob/updateJob Server Action | All job field input + FK references are untrusted; Zod-validated, FK-guarded, tenant-scoped. |
| client dropdown → transitionJobStatusAction | Status change requests validated server-side by transitionJobStatus (UI filtering is UX-only). |
| client line-item amounts → totals | Money strings are server-recomputed by computeJobTotals; client totals are display-only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-01 | Tampering | Status change via UI | mitigate | Dropdown filtering is UX-only (D-04); the server transitionJobStatus is the enforcement point; updateJob excludes status |
| T-3-02 | Information Disclosure / Elevation | createJob/updateJob FK refs | mitigate | Cross-tenant guard on customerId/contactId/serviceLocationId/categoryId/jobSourceId/taxItemId before insert (catalog guard pattern); withTenant + RLS |
| T-3-03 | Tampering | Line-item money + client totals | mitigate | Server-authoritative computeJobTotals; Zod-validate numeric money strings; never persist client-sent totals |
| T-3-05 | Input Validation | All job + line-item fields | mitigate | Zod safeParse on every action input; enum membership for billingType/lineItemType |
| T-3-SC | Tampering | npm/pip/cargo installs | accept | No package installs (reuses installed Base UI, nuqs, etc.) |
</threat_model>
