<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Square CDN | SDK loaded over HTTPS; tokenization happens in the browser |
| PWA → Square payment server action | Card token (sourceId) crosses; server authorizes with server-only token and writes ledger |
| PWA → Phase 7/8 invoice/send actions | Tech-submitted invoice/send payload crosses; server enforces tenant + role + per-trigger settings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Tampering | Square card token handling | mitigate | Token posted only to a dedicated audited squarePaymentAction over HTTPS (not a generic action); server uses SQUARE_ACCESS_TOKEN (server-only) to authorize; SDK loaded over HTTPS (RESEARCH Pitfall 5) |
| T-05-06b | Information Disclosure | Square access token exposure | mitigate | SQUARE_ACCESS_TOKEN is server-only, never NEXT_PUBLIC; only application/location IDs are client-exposed |
| T-05-18 | Tampering | Ledger dedup/audit | mitigate | Square payment posts to the canonical Phase 7 ledger writer with dedup + audit trail (D-22); the PWA does not write ledger rows directly |
| T-05-19 | Information Disclosure | Invoice cache cross-tenant | mitigate | invoices table in the per-org Dexie DB, carries tenantId, cleared on sign-out |
| T-05-20 | Repudiation | Offline payment integrity | mitigate | Square card payments are online-only (D-03), never queued; only cash/check manual notes queue, which are non-authoritative records confirmed server-side on sync |
| T-05-SC | Tampering | Square CDN script | accept | Square Web Payments SDK has no npm package; loaded from official Square CDN over HTTPS per RESEARCH; documented external runtime SDK, not a persisted dependency |
</threat_model>
