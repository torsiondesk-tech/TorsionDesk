<objective>
Deliver the first end-to-end technician runtime slice: the date-grouped 7-day schedule and the offline-capable status transition flow. A technician can open the app, see their jobs grouped by day, open a job, tap Update Status, choose a legal next status, and have that change persist online or queue offline and auto-sync.

Purpose: Status updates are the single most frequent field action. This slice makes the PWA immediately useful end-to-end (schedule → job → status → sync) before photos/signatures/estimates/invoices layer on.

Output: A working /tech/jobs schedule, a job detail page with the status card + bottom sheet, the Dexie-backed sync loop with offline queueing and FIFO flush, and the online/offline/pending indicator.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PWA client → server action | Tech-submitted status/jobId crosses; server FSM + RLS re-validate |
| Dexie outbox → server | Queued mutations replayed after reconnect; session must still be valid |
| Realtime broadcast → Dexie cache | Inbound job-updated events refresh the local cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03 | Tampering | Job status write path | mitigate | All status changes call transitionJobStatusAction → transitionJobStatus → FSM isLegalTransition server-side; the bottom sheet only offers ALLOWED_TRANSITIONS but the server is the enforcer (D-21) |
| T-05-09 | Tampering / Elevation | Outbox replay after sign-out | mitigate | flushOutbox runs inside the authenticated session; server action re-reads Clerk orgId/userId and rejects if absent; failed illegal transitions are surfaced not silently retried (Pitfall 3) |
| T-05-10 | Information Disclosure | Schedule shows only own jobs | mitigate | listJobs called with assigneeUserId: userId and withTenant(orgId); no cross-tenant or cross-tech leakage |
| T-05-11 | Denial of Service | Infinite retry on illegal transition | mitigate | flushOutbox caps/handles "Illegal transition" by marking failed and surfacing to the tech instead of looping |
</threat_model>
