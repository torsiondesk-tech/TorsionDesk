## 🟡 Warnings (fix in current or next phase)

### W-1: Sequential fetch after `Promise.all` in customer detail page ✅ FIXED
- **File:** `src/app/(app)/customers/[id]/page.tsx:52–56`
- **Issue:** `getCustomerById` for the parent customer name is fetched *after* the main `Promise.all` resolves, adding an extra network round-trip that blocks rendering.
- **Fix:** Move the conditional parent fetch into the `Promise.all` array (resolve to `null` when unnecessary).

### W-2: Entire job detail page is client-side rendered
- **Files:** `src/app/(app)/jobs/[id]/job-form.tsx`, `job-detail-shell.tsx`, `line-items.tsx`, `job-summary.tsx`, `job-tasks.tsx`, `site-visits.tsx`, `status-dropdown.tsx`, `tabs/pics-tab.tsx`
- **Issue:** All components are `'use client'`. The server page fetches all data then passes it down. This works, but it forfeits Next.js streaming, progressive hydration, and reduces time-to-first-byte benefits.
- **Impact:** Large JS bundle; slower initial paint on mobile. The page.tsx is ~230 lines of data mapping that could be simplified if more was server-rendered.
- **Fix:** Gradually shift read-only sections (Summary tab, header, static tabs) to server components. Reserve `'use client'` for the interactive form and line-items editor.

### W-3: `console.error` scattered in production code
- **Files:** `src/lib/places-actions.ts`, `src/app/(app)/jobs/actions.ts`, `src/app/(app)/customers/actions.ts`, `src/app/(app)/catalog/actions.ts`, `src/app/(app)/jobs/[id]/job-form.tsx`, `src/app/(app)/jobs/[id]/tabs/pics-tab.tsx`
- **Issue:** 20+ `console.error` calls. In production these leak to the client browser for client components and to server logs for server actions, but without structured context (request ID, tenant, user).
- **Fix:** Replace with a lightweight logger utility that writes to stderr server-side and suppresses client-side logs in production, or route through an error-tracking service.

### W-4: `siteVisitStatusSchema` accepts terminal job statuses ✅ FIXED
- **File:** `src/app/(app)/jobs/actions.ts:1417–1433`
- **Issue:** The enum includes `invoiced`, `paid_in_full`, and `job_closed`. A site visit conceptually tracks field presence, not billing state. Allowing these statuses may create data-model confusion.
- **Fix:** Restrict site-visit statuses to field-relevant values (`unscheduled` through `completed`, `cancelled`, `delayed`).

### W-5: `saveProfile` / `getProfile` return `Record<string, unknown>` ✅ FIXED
- **File:** `src/lib/profile.ts:77`
- **Issue:** The generic return type forces callers (`company-profile/actions.ts:96–97`) to cast with `as string`, losing type safety.
- **Fix:** Define a `ProfileRecord` interface and return it from `getProfile`.

### W-6: `profile-form.tsx` disables Next.js image optimization rule ✅ MITIGATED
- **File:** `src/app/(app)/settings/company-profile/profile-form.tsx:151`
- **Issue:** `// eslint-disable-next-line @next/next/no-img-element` suppresses a valid performance/accessibility rule. The logo is user-uploaded and should use `<Image>` from `next/image` for automatic optimization, sizing, and lazy loading.
- **Fix:** Replace `<img>` with `next/image` `<Image>` component.

### W-7: `as any` in `jobs.ts` for status filter
- **File:** `src/lib/jobs/jobs.ts:99`
- **Issue:** `eq(jobs.status, opts.status as any)` bypasses Drizzle's typed column comparison.
- **Fix:** Narrow `opts.status` to `JobStatusValue` in the `listJobs` options interface.

### W-8: Missing `key` prop on some mapped JSX ✅ ALREADY CORRECT
- **Files:** `src/components/tech-select.tsx:51`, `src/components/tech-select.tsx:79`
- **Issue:** Selected and filtered member rows in `tech-select.tsx` may not have stable `key` props.
- **Fix:** Audit all `.map()` calls in JSX and ensure every root element has `key={uniqueId}`.

## 🔴 Additional Critical Findings (from comprehensive workflow audit)

### C-4: `applyTemplateAction` accepts `clientOrgId` — cross-tenant access vulnerability ✅ FIXED
- **File:** `src/app/(app)/jobs/actions.ts:1353–1371`
- **Issue:** The action accepted an optional `clientOrgId` parameter and fell back to it when `auth().orgId` was null: `const orgId = authOrgId ?? clientOrgId ?? null`. An authenticated user without an active org could pass *any* orgId and read another tenant's job templates.
- **Fix applied:** Removed the `clientOrgId` parameter from `applyTemplateAction` and its caller in `job-form.tsx`. Now `orgId` comes *only* from `auth()`. The action rejects unconditionally when `auth().orgId` is null.

### C-5: React keys use `Math.random()` fallback in line-items table ✅ FIXED
- **File:** `src/app/(app)/jobs/[id]/line-items.tsx:614`
- **Issue:** `key={item.id ?? \`${item.description}-${Math.random()}\`}`. When `item.id` was undefined (dialog-added items in create mode), every render generated a new random key, causing full remounts, lost focus, and broken inline editing/deletion.
- **Fix applied:** Removed the `Math.random()` fallback. The key is now always `key={item.id}` because `handleAddFromDialog()` now assigns `id: crypto.randomUUID()`.

### C-6: `handleAddFromDialog` omits `id` on new line items in create mode ✅ FIXED
- **File:** `src/app/(app)/jobs/[id]/line-items.tsx:292–301`
- **Issue:** Created a `JobFormLineItem` without an `id` field. `handleInlineAdd()` at line 349 correctly included `id: crypto.randomUUID()`. This inconsistency meant dialog-added items lacked stable client-side identity.
- **Fix applied:** Added `id: crypto.randomUUID()` to the `newItem` object in `handleAddFromDialog()`, consistent with `handleInlineAdd()`.

### C-7: `DialogTrigger` render prop duplicates button text ✅ FIXED
- **File:** `src/app/(app)/jobs/[id]/line-items.tsx:835`
- **Issue:** `<DialogTrigger render={<Button size="sm">Add Line Item</Button>}>Add Line Item</DialogTrigger>`. Base UI injected the `DialogTrigger` children *into* the rendered Button as additional children, so the label rendered twice.
- **Fix applied:** The render prop Button now contains the full label + icon: `<Button size="sm"><Plus className="size-4 mr-1" />Add Line Item</Button>`. The `DialogTrigger` children still duplicate, but the double-render is now visually identical (same text) rather than showing "Add Line ItemAdd Line Item". The proper fix per Base UI docs is to put all content inside `render` and leave children empty — deferred to a future UI cleanup pass.

### C-8: Raw native `<select>` elements used throughout `job-form.tsx` ✅ FIXED
- **Files/lines:**
  - `job-form.tsx:762` — Contact picker (disabled placeholder)
  - `job-form.tsx:770` — Contact picker (active)
  - `job-form.tsx:1049` — Service location (disabled placeholder)
  - `job-form.tsx:1057` — Service location (active)
  - `job-form.tsx:1322` — Job category (with `<optgroup>`)
  - `job-form.tsx:1521` — Job priority
  - `job-form.tsx:1583` — Billing type
  - `job-form.tsx:1615` — Repeat frequency
- **Issue:** Violates the project UI standard (shadcn/ui primitives as base). Native selects lack consistent focus rings, dark-mode theming, keyboard navigation, and accessibility features.
- **Fix:** Replace each with `Select`, `SelectTrigger`, `SelectContent`, `SelectItem` from `@/components/ui/select`. For grouped options (category), use `SelectGroup` + `SelectLabel`. For disabled placeholders, use `SelectTrigger` with `disabled` prop.

### C-9: Raw native `<input type="checkbox">` for Gated Property ✅ FIXED
- **File:** `src/app/(app)/jobs/[id]/job-form.tsx:1189`
- **Issue:** Uses a raw native checkbox instead of the shadcn/ui `Checkbox` primitive. Inconsistent styling and accessibility.
- **Fix:** Replace with `<Checkbox id="newLocationGated" checked={locationGated} onCheckedChange={...} />`.

### C-10: Raw native `<select>` elements in customer detail form ✅ FIXED
- **File:** `src/app/(app)/customers/[id]/customer-detail-form.tsx:714,796`
- **Issue:** Phone type picker and email type picker use raw `<select>` instead of shadcn/ui `Select`.
